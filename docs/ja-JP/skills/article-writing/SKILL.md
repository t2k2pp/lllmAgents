# 記事ライティング

汎用的なAI出力ではなく、実在の人物やブランドのように聞こえる長文コンテンツを執筆する。

## 発動条件

- ブログ記事、エッセイ、ローンチ投稿、ガイド、チュートリアル、またはニュースレターの執筆
- メモ、トランスクリプト、リサーチ資料を洗練された記事に仕上げる
- 例文から既存の創業者、運営者、またはブランドの文体を再現する
- すでに書かれた長文コピーの構成、ペーシング、根拠を改善する

## 基本ルール

1. 具体的なものからリードする: 例、出力、エピソード、数字、スクリーンショットの説明、またはコードブロック。
2. 例の後に説明する。前には説明しない。
3. 冗長な文よりも短く直接的な文を好む。
4. 利用可能で出典のある具体的な数字を使う。
5. 経歴上の事実、企業の指標、顧客の証言を捏造しない。

## 文体キャプチャワークフロー

ユーザーが特定の文体を求める場合、以下の1つ以上を収集する:
- 公開記事
- ニュースレター
- X / LinkedIn投稿
- ドキュメントやメモ
- 簡単なスタイルガイド

その後、以下を抽出する:
- 文の長さとリズム
- フォーマル、会話調、鋭いのいずれか
- 好まれる修辞手法（括弧、リスト、断片的な文、質問など）
- ユーモア、意見、逆張りフレーミングの許容度
- 見出し、箇条書き、コードブロック、引用などのフォーマット習慣

文体の参考が提供されない場合は、直接的で実務者スタイルの文体をデフォルトとする: 具体的、実践的、誇大表現を控えめに。

## 禁止パターン

以下が見つかった場合、削除して書き直す:
- 「急速に進化する今日の環境において」のような汎用的な書き出し
- 「さらに」「加えて」のような場つなぎの接続詞
- 「ゲームチェンジャー」「最先端」「革命的」のような誇大フレーズ
- 根拠のない曖昧な主張
- 提供されたコンテキストに裏付けのない経歴や信頼性に関する主張

## 執筆プロセス

1. 読者と目的を明確にする。
2. 各セクションに1つの目的を持つ骨組みのアウトラインを構築する。
3. 各セクションを証拠、例、またはシーンから始める。
4. 次の文がその場所に値する場合にのみ展開する。
5. テンプレート的または自画自賛的に聞こえるものを削除する。

## 構成ガイダンス

### 技術ガイド
- 読者が得られるものから始める
- すべての主要セクションにコードまたはターミナルの例を使用する
- やわらかい要約ではなく、具体的なポイントで締めくくる

### エッセイ / 意見記事
- 緊張感、矛盾、または鋭い観察から始める
- セクションごとに1つの議論スレッドを維持する
- 意見を裏付ける例を使用する

### ニュースレター
- 最初の画面を強くする
- 日記のような埋め合わせではなく、洞察とアップデートを混ぜる
- 明確なセクションラベルとスキャンしやすい構造を使用する

## 品質ゲート

納品前に:
- 提供された情報源に対して事実の主張を検証する
- 場つなぎや企業的な言い回しを削除する
- 文体が提供された例と一致していることを確認する
- すべてのセクションが新しい情報を追加していることを確認する
- 対象プラットフォームに合わせたフォーマットを確認する
