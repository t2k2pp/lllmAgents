# フロントエンドスライド

依存関係ゼロで、アニメーションリッチなHTMLプレゼンテーションをブラウザ上で完全に動作する形で作成する。

[zarazhangrui](https://github.com/zarazhangrui)が示したビジュアル探索アプローチにインスパイアされている。

## 発動条件

- トークデッキ、ピッチデッキ、ワークショップデッキ、社内プレゼンテーションの作成
- `.ppt`または`.pptx`スライドをHTMLプレゼンテーションに変換
- 既存のHTMLプレゼンテーションのレイアウト、モーション、タイポグラフィの改善
- デザインの好みがまだわからないユーザーとプレゼンテーションスタイルを探索

## 絶対条件

1. **依存関係ゼロ**: デフォルトでインラインCSSとJSを含む1つの自己完結型HTMLファイル。
2. **ビューポートフィットは必須**: すべてのスライドが内部スクロールなしで1つのビューポートに収まること。
3. **見せて、語らない**: 抽象的なスタイルアンケートではなくビジュアルプレビューを使用する。
4. **独自性のあるデザイン**: 汎用的な紫グラデーション、Inter-on-white、テンプレートのようなデッキを避ける。
5. **プロダクション品質**: コメント付き、アクセシブル、レスポンシブ、パフォーマントなコードを維持する。

生成前に、ビューポートセーフなCSSベース、密度制限、プリセットカタログ、CSSの注意事項については`STYLE_PRESETS.md`を読むこと。

## ワークフロー

### 1. モードの検出

1つのパスを選択する:
- **新規プレゼンテーション**: ユーザーがトピック、メモ、または完全な原稿を持っている
- **PPT変換**: ユーザーが`.ppt`または`.pptx`を持っている
- **改善**: ユーザーがすでにHTMLスライドを持っていて改善を希望

### 2. コンテンツの把握

最低限必要なことだけを聞く:
- 目的: ピッチ、教育、カンファレンストーク、社内アップデート
- 長さ: 短い（5-10）、中程度（10-20）、長い（20+）
- コンテンツの状態: 完成原稿、ラフメモ、トピックのみ

ユーザーがコンテンツを持っている場合、スタイリング前に貼り付けてもらう。

### 3. スタイルの探索

デフォルトでビジュアル探索を行う。

ユーザーが希望するプリセットをすでに知っている場合、プレビューをスキップして直接使用する。

それ以外の場合:
1. デッキが作り出すべき印象を聞く: 感銘、活力、集中、インスピレーション。
2. `.ecc-design/slide-previews/`に**3つの単一スライドプレビューファイル**を生成する。
3. 各プレビューは自己完結型で、タイポグラフィ/色/モーションを明確に示し、スライドコンテンツはおよそ100行以下に収める。
4. ユーザーにどのプレビューを採用するか、またはどの要素をミックスするかを聞く。

ムードからスタイルへのマッピングには`STYLE_PRESETS.md`のプリセットガイドを使用する。

### 4. プレゼンテーションの構築

以下のいずれかを出力する:
- `presentation.html`
- `[presentation-name].html`

抽出された画像やユーザー提供の画像がデッキに含まれる場合のみ`assets/`フォルダを使用する。

必須構造:
- セマンティックなスライドセクション
- `STYLE_PRESETS.md`からのビューポートセーフなCSSベース
- テーマ値のためのCSSカスタムプロパティ
- キーボード、ホイール、タッチナビゲーション用のプレゼンテーションコントローラークラス
- 表示アニメーションのトリガー用Intersection Observer
- reduced-motionサポート

### 5. ビューポートフィットの強制

これはハードゲートとして扱う。

ルール:
- すべての`.slide`は`height: 100vh; height: 100dvh; overflow: hidden;`を使用すること
- すべてのテキストとスペーシングは`clamp()`でスケールすること
- コンテンツが収まらない場合、複数のスライドに分割すること
- 読めないサイズまでテキストを縮小してオーバーフローを解決しないこと
- スライド内にスクロールバーを許可しないこと

密度制限と必須CSSブロックは`STYLE_PRESETS.md`を使用する。

### 6. 検証

完成したデッキを以下のサイズで確認する:
- 1920x1080
- 1280x720
- 768x1024
- 375x667
- 667x375

ブラウザオートメーションが利用可能な場合、スライドのオーバーフローがなくキーボードナビゲーションが機能することを検証する。

### 7. 納品

引き渡し時に:
- ユーザーが保持を希望しない限り、一時プレビューファイルを削除する
- 有用な場合、プラットフォーム適切なオープナーでデッキを開く
- ファイルパス、使用したプリセット、スライド数、簡単なテーマカスタマイズポイントをまとめる

現在のOSに応じた正しいオープナーを使用する:
- macOS: `open file.html`
- Linux: `xdg-open file.html`
- Windows: `start "" file.html`

## PPT / PPTX変換

PowerPoint変換の場合:
1. テキスト、画像、ノートの抽出には`python3`と`python-pptx`を優先する。
2. `python-pptx`が利用できない場合、インストールするか手動/エクスポートベースのワークフローにフォールバックするかを聞く。
3. スライドの順序、スピーカーノート、抽出されたアセットを保持する。
4. 抽出後、新規プレゼンテーションと同じスタイル選択ワークフローを実行する。

変換はクロスプラットフォームに保つ。Pythonでできる場合、macOS専用ツールに依存しない。

## 実装要件

### HTML / CSS

- ユーザーが明示的にマルチファイルプロジェクトを希望しない限り、インラインCSSとJSを使用する。
- フォントはGoogle FontsまたはFontshareから取得可能。
- 雰囲気のある背景、強いタイプ階層、明確なビジュアル方向を好む。
- イラストレーションではなく、抽象的な形状、グラデーション、グリッド、ノイズ、ジオメトリを使用する。

### JavaScript

以下を含める:
- キーボードナビゲーション
- タッチ / スワイプナビゲーション
- マウスホイールナビゲーション
- プログレスインジケータまたはスライドインデックス
- 表示時アニメーショントリガー

### アクセシビリティ

- セマンティック構造を使用する（`main`、`section`、`nav`）
- コントラストを読みやすく保つ
- キーボードのみのナビゲーションをサポートする
- `prefers-reduced-motion`を尊重する

## コンテンツ密度制限

ユーザーが明示的に高密度のスライドを要求し、読みやすさが維持される場合を除き、以下の最大値を使用する:

| スライドタイプ | 制限 |
|--------------|------|
| タイトル | 見出し1つ + サブタイトル1つ + オプションのタグライン |
| コンテンツ | 見出し1つ + 箇条書き4-6個または短い段落2つ |
| フィーチャーグリッド | 最大6カード |
| コード | 最大8-10行 |
| 引用 | 引用1つ + 帰属 |
| 画像 | ビューポートで制約された画像1つ |

## アンチパターン

- ビジュアルアイデンティティのない汎用的なスタートアップグラデーション
- 意図的にエディトリアルでない限り、システムフォントのデッキ
- 長い箇条書きの壁
- スクロールが必要なコードブロック
- 短い画面で壊れる固定高さのコンテンツボックス
- `-clamp(...)`のような無効な否定CSSファンクション

## 納品チェックリスト

- プレゼンテーションがブラウザでローカルファイルから実行できる
- すべてのスライドがスクロールなしでビューポートに収まる
- スタイルが独自性があり意図的である
- アニメーションが意味を持ち、うるさくない
- reduced motionが尊重されている
- ファイルパスとカスタマイズポイントが引き渡し時に説明されている
