# 開発ワークフロー

> このファイルは [common/git-workflow.md](./git-workflow.md) を、git 操作の前に行われる完全な機能開発プロセスで拡張します。

機能実装ワークフローは、開発パイプラインを記述します: 調査、計画、TDD、コードレビュー、そして git へのコミット。

## 機能実装ワークフロー

0. **調査と再利用** _（新しい実装の前に必須）_
   - **まず GitHub コード検索:** 新しいコードを書く前に、`gh search repos` と `gh search code` を実行して、既存の実装、テンプレート、パターンを見つけてください。
   - **調査に Exa MCP を使用:** 計画フェーズで `exa-web-search` MCP を使用して、より広範な調査、データ取り込み、先行事例の発見を行ってください。
   - **パッケージレジストリを確認:** ユーティリティコードを書く前に npm、PyPI、crates.io、その他のレジストリを検索してください。手作りのソリューションよりも実戦テスト済みのライブラリを優先してください。
   - **適応可能な実装を検索:** 問題の 80% 以上を解決し、フォーク、移植、またはラップできるオープンソースプロジェクトを探してください。
   - 要件を満たす場合は、ゼロから新しいコードを書くよりも、実証済みのアプローチを採用または移植することを優先してください。

1. **まず計画**
   - **planner** agent を使用して実装計画を作成
   - コーディング前に計画ドキュメントを生成: PRD、アーキテクチャ、システム設計、技術文書、タスクリスト
   - 依存関係とリスクを特定
   - フェーズに分割

2. **TDD アプローチ**
   - **tdd-guide** agent を使用
   - まずテストを書く（RED）
   - テストをパスするように実装（GREEN）
   - リファクタリング（IMPROVE）
   - 80%+ カバレッジを確認

3. **コードレビュー**
   - コード記述直後に **code-reviewer** agent を使用
   - CRITICAL と HIGH の問題に対処
   - 可能な限り MEDIUM の問題を修正

4. **コミット & プッシュ**
   - 詳細なコミットメッセージ
   - Conventional Commits フォーマットに従う
   - コミットメッセージのフォーマットと PR プロセスについては [git-workflow.md](./git-workflow.md) を参照
