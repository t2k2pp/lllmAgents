# /learn-eval - 抽出、評価、保存

`/learn` を拡張し、スキルファイルを書き込む前に品質ゲートと保存場所の決定を追加します。

## 抽出する内容

以下を探します:

1. **エラー解決パターン** — 根本原因 + 修正 + 再利用可能性
2. **デバッグ技術** — 自明ではない手順、ツールの組み合わせ
3. **回避策** — ライブラリの癖、API の制限、バージョン固有の修正
4. **プロジェクト固有のパターン** — 規約、アーキテクチャの決定、統合パターン

## プロセス

1. セッションで抽出可能なパターンをレビュー
2. 最も価値がある/再利用可能な洞察を特定

3. **保存場所を決定:**
   - 「このパターンは別のプロジェクトでも役立つか？」と自問する
   - **グローバル** (`~/.claude/skills/learned/`): 2つ以上のプロジェクトで使える汎用パターン（bash の互換性、LLM API の動作、デバッグ技術など）
   - **プロジェクト** (現在のプロジェクトの `.claude/skills/learned/`): プロジェクト固有の知識（特定の設定ファイルの癖、プロジェクト固有のアーキテクチャ決定など）
   - 迷った場合はグローバルを選択（グローバル → プロジェクトへの移動はその逆より容易）

4. 以下の形式でスキルファイルを下書き:

```markdown
---
name: pattern-name
description: "130文字以内"
user-invocable: false
origin: auto-extracted
---

# [説明的なパターン名]

**抽出日:** [日付]
**コンテキスト:** [いつ適用されるかの簡単な説明]

## 問題
[解決する問題 - 具体的に]

## 解決策
[パターン/技術/回避策 - コード例付き]

## 使用タイミング
[トリガー条件]
```

5. **保存前に以下のルーブリックで自己評価:**

   | 次元 | 1 | 3 | 5 |
   |------|---|---|---|
   | 具体性 | 抽象的な原則のみ、コード例なし | 代表的なコード例あり | すべての使用パターンをカバーする豊富な例 |
   | 実行可能性 | 何をすべきか不明確 | 主要な手順は理解可能 | すぐに実行可能、エッジケースもカバー |
   | スコープ適合性 | 広すぎるか狭すぎる | ほぼ適切だが境界に曖昧さあり | 名前、トリガー、内容が完全に一致 |
   | 非冗長性 | 別のスキルとほぼ同一 | 一部重複するが独自の視点あり | 完全にユニークな価値 |
   | カバレッジ | 対象タスクのごく一部のみカバー | 主要ケースはカバー、一般的なバリアントが欠落 | 主要ケース、エッジケース、落とし穴をカバー |

   - 各次元を 1-5 で採点
   - いずれかの次元が 1-2 の場合、すべての次元が 3 以上になるまで下書きを改善して再採点
   - ユーザーにスコア表と最終下書きを表示

6. ユーザーに確認を求める:
   - 表示: 提案された保存パス + スコア表 + 最終下書き
   - 書き込み前に明示的な確認を待つ

7. 決定された場所に保存

## ステップ 5 の出力形式（スコア表）

| 次元 | スコア | 根拠 |
|------|--------|------|
| 具体性 | N/5 | ... |
| 実行可能性 | N/5 | ... |
| スコープ適合性 | N/5 | ... |
| 非冗長性 | N/5 | ... |
| カバレッジ | N/5 | ... |
| **合計** | **N/25** | |

## 注意事項

- 些細な修正（タイプミス、単純な構文エラー）は抽出しない
- 一度限りの問題（特定の API の障害など）は抽出しない
- 将来のセッションで時間を節約できるパターンに焦点を当てる
- スキルは集中させる — 1つのスキルに1つのパターン
- カバレッジスコアが低い場合、保存前に関連するバリアントを追加する
