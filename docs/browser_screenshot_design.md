# ブラウザスクリーンショット保存に関する設計書

## 1. 背景と課題
ユーザーから「ブラウザのスクリーンショットを保存するよう指示したところ、ファイルの前半（base64文字列の一部）しか保存されなかった」という事象が報告された。
調査の結果、現在Agentが使用している `browser_screenshot` ツールは、取得したスクリーンショットの画像容量によるコンテキスト超過を防ぐため、以下のような仕様になっていることが判明した。
* スクリーンショット撮影後、バッファをbase64エンコードし、最初の100文字のみを切り出してAgentに返却している。
* その結果、Agentがスクリーンショットを保存するよう指示を受けた際、切り出された不完全なテキスト（`iVBORw...`）をそのままファイルとして書き込んでしまっていた。

## 2. 対策方針
画像全体の巨大なbase64文字列をAgentに返却するのは、トークン長およびコンテキストの圧迫により適切ではない。
そこで、**ツール内で直接ファイルへの書き込みが行える仕組み**を導入する。

### 具体的な変更点
`src/tools/definitions/browser.ts` に定義されている `browser_screenshot` ツールの引数（パラメータ）を拡張する。
* **追加するパラメータ**: `save_path` (string, 任意)
* **挙動の変更**:
  1. `browser_screenshot` の実行時、`save_path` が指定されている場合は、取得したバッファを `fs.promises.writeFile` を用いて直接指定先のパス（ローカルファイル）に保存する。
  2. 保存後、Agentには「指定パスに保存が完了した」旨のメッセージを返す（不要なbase64は返さない）。
  3. `save_path` が指定されていない場合（内部解析など）は、従来通り先頭100文字を返す後方互換を維持する。

## 3. 修正対象ファイル
* `src/tools/definitions/browser.ts`

## 4. 期待される効果
* LLM（Agent）が巨大な画像データを仲介することなく、直接画像をローカルに保存できるようになり、完全なPNGファイルが作成可能になる。
* コンテキスト長の超過や文字切れエラーを回避できる。
